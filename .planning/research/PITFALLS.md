# Domain Pitfalls

**Domain:** TypeScript SDK — identity verification, presigned S3 uploads, cross-runtime, dual ESM/CJS, pnpm monorepo
**Project:** deepidv Server SDK (`@deepidv/server`)
**Researched:** 2026-04-05
**Confidence:** HIGH (these are well-established, reproducible pitfalls across the TypeScript SDK ecosystem)

---

## Critical Pitfalls

Mistakes that cause rewrites, broken npm installs for consumers, silent runtime failures, or security issues.

---

### Pitfall 1: Missing or Wrong `exports` Field in package.json

**What goes wrong:** Publishing packages without a proper `"exports"` map means Node 12+ ESM consumers hit `ERR_PACKAGE_PATH_NOT_EXPORTED`, CJS consumers get the wrong file, and sub-path imports break entirely. tsup generates dual output files (`.mjs` / `.cjs`) but they are dead unless `package.json` routes to them correctly.

**Why it happens:** Developers look at the `"main"` and `"module"` fields from the pre-exports era and assume they are sufficient. They are not for Node 12+ ESM resolution.

**Consequences:** Published package that appears to install correctly but throws at import time. CJS callers get `exports is not defined` or lose tree-shaking. ESM callers get `ERR_PACKAGE_PATH_NOT_EXPORTED`. TypeScript consumers get `Cannot find module` even though types exist on disk.

**Prevention:**

Every published package needs this exact shape:

```json
{
  "exports": {
    ".": {
      "import": {
        "types": "./dist/index.d.mts",
        "default": "./dist/index.mjs"
      },
      "require": {
        "types": "./dist/index.d.cts",
        "default": "./dist/index.cjs"
      }
    }
  },
  "main": "./dist/index.cjs",
  "module": "./dist/index.mjs",
  "types": "./dist/index.d.ts"
}
```

Keep `"main"` and `"types"` as fallbacks for tooling that does not read `"exports"`. The `"types"` condition inside `"exports"` must come before `"default"` — TypeScript resolves conditions in order.

**Warning signs:**
- `ERR_PACKAGE_PATH_NOT_EXPORTED` in consumer projects
- `Cannot find module '@deepidv/server'` with `moduleResolution: bundler`
- CJS consumers receive `.mjs` files (esbuild/webpack warnings about format mismatch)
- `tsconfig` with `moduleResolution: node16` or `bundler` starts complaining

**Phase:** Phase 1 (monorepo scaffolding). Get this right before writing any module code. A broken exports map is invisible until a consumer tries to install.

### Pitfall 2: Dual Declaration Files — `.d.ts` vs `.d.mts` vs `.d.cts`

**What goes wrong:** tsup v6+ generates `.d.mts` for ESM output and `.d.cts` for CJS output alongside the old `.d.ts`. If `"types"` in `package.json` points to `.d.ts` but the consumer's `moduleResolution` is `node16` or `bundler`, TypeScript picks up the wrong declaration file and type imports fail or produce incorrect types.

**Why it happens:** Many tutorials pre-date the `.d.mts`/`.d.cts` split. Copy-pasting old config skips this.

**Consequences:** TypeScript consumers using `moduleResolution: node16` or `bundler` (the correct modern setting) get type errors on correct code. Worse: they get no errors on incorrect code because TypeScript silently falls back to unresolved `any`.

**Prevention:**

In `tsup.config.ts`, enable both declaration file variants:

```typescript
export default defineConfig({
  entry: ['src/index.ts'],
  format: ['cjs', 'esm'],
  dts: true,
  // tsup >= 7.x generates .d.mts and .d.cts automatically with format splitting
});
```

In `package.json` `"exports"`, explicitly map `"types"` for both conditions (as shown in Pitfall 1). Test by installing the package in a consumer project with `moduleResolution: bundler` and running `tsc --noEmit`.

**Warning signs:**
- `TS2305: Module '"@deepidv/server"' has no exported member 'DeepIDV'` in consuming projects
- Hover in VS Code shows `any` instead of the actual type
- `tsc --noEmit` passes but `moduleResolution: node16` version fails

**Phase:** Phase 1. Verify declaration output before moving to Phase 2.

### Pitfall 3: S3 PUT Requires Exact Content-Type — No Mismatch Tolerated

**What goes wrong:** The presigned URL is generated by the deepidv API with a specific `Content-Type` embedded in the URL signature. If the SDK sends a different `Content-Type` header on the PUT (or omits it entirely), S3 returns `403 SignatureDoesNotMatch` — not a helpful error. Developers see a 403 and assume their API key is wrong.

**Why it happens:** Developers default to `Content-Type: application/octet-stream` or let `fetch` pick the type automatically. The signed URL was generated for `image/jpeg`. The signature check fails.

**Consequences:** All file uploads fail with opaque 403 errors. The error message (`SignatureDoesNotMatch`) is returned as XML in the response body, not JSON, so any JSON-parsing error handler makes the situation worse. Developer support tickets flood in about "invalid API key."

**Prevention:**

1. The presign request to deepidv API must include `contentType` per file. The SDK detects content type before requesting the presigned URL.
2. The S3 PUT must send `Content-Type` header that exactly matches what was signed.
3. The content-type detection must be reliable — not just file extension sniffing.
4. S3 PUT response body is XML on error. Parse it as text, not JSON, and surface the `<Code>` element.

Content-type detection strategy (in order of reliability):
- If caller provides explicit `mimeType` option → use that
- If input is a file path with `.jpg`/`.jpeg`/`.png`/`.webp`/`.pdf` extension → map to MIME type
- If input is a Buffer/Uint8Array → read magic bytes (JPEG: `FF D8 FF`, PNG: `89 50 4E 47`, PDF: `25 50 44 46`)
- Default to `image/jpeg` for identity verification use cases (but log a warning)

**Warning signs:**
- S3 PUT returning 403 with XML body containing `SignatureDoesNotMatch`
- Upload works in dev with JPEG files but breaks with PNG or WebP inputs
- Error handler receiving XML string where it expected JSON

**Phase:** Phase 2 (presigned upload handler). This is the highest-risk single piece of code in the SDK.

### Pitfall 4: ReadableStream is Consumed Once — Double-Read Fails Silently

**What goes wrong:** `identity.verify()` accepts a `documentImage` and a `faceImage`, both of which can be `ReadableStream`. If the same stream is accidentally passed for both, or if the SDK reads it once to detect content type and again to upload, the second read returns nothing (empty body). S3 receives a zero-byte file. The upload "succeeds" (S3 accepts empty files on presigned URLs in some configurations), but the processing endpoint returns a garbled or empty result.

**Why it happens:** Streams are lazy and single-use. Code that reads a Buffer naturally can read it twice. Code that reads a ReadableStream cannot. Calling `stream.getReader().read()` for magic-byte detection drains the stream for the upload.

**Consequences:** Silent data corruption. The SDK call completes with no error, but the OCR or face detection result is wrong/empty. Extremely hard to debug.

**Prevention:**

1. Never read a ReadableStream to detect content type — require the caller to provide `mimeType` when passing a stream, or require them to pass a Buffer instead.
2. When receiving a ReadableStream, convert it to a `Uint8Array` once at the SDK boundary (in the upload handler), then work from the buffer for all subsequent operations.
3. Add a runtime check: if the caller passes the same stream reference for `source` and `target` in `face.compare()`, throw immediately with a descriptive error.

Conversion pattern:
```typescript
async function streamToBuffer(stream: ReadableStream<Uint8Array>): Promise<Uint8Array> {
  const reader = stream.getReader();
  const chunks: Uint8Array[] = [];
  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    chunks.push(value);
  }
  const total = chunks.reduce((acc, c) => acc + c.length, 0);
  const result = new Uint8Array(total);
  let offset = 0;
  for (const chunk of chunks) {
    result.set(chunk, offset);
    offset += chunk.length;
  }
  return result;
}
```

**Warning signs:**
- OCR or face detection returns empty/garbage results with no error
- S3 access logs show zero-byte PUTs that return 200
- Tests with Buffer inputs pass; tests with ReadableStream inputs produce wrong results

**Phase:** Phase 2 (upload handler). Phase 3 (face.compare parallel uploads) for the double-read variant.

### Pitfall 5: Retry Logic Retrying Non-Retriable Errors

**What goes wrong:** Retry logic that retries all errors (including 4xx) hammers the API, wastes quota, and delays the error surface to the developer. Worse: retrying a `ValidationError` (400) on a malformed input never recovers — it just burns 3 retries before the developer sees the error they should have seen immediately.

**Why it happens:** Generic "retry on failure" implementations catch all exceptions or all non-200 responses. The edge case of 4xx-vs-5xx is easy to overlook.

**Consequences:** 400 errors take 3x as long to surface (exponential backoff makes this even worse). 401 errors result in three API key verification attempts. 422 validation errors are delayed. Developer thinks the SDK is hanging.

**Prevention:**

Retry only on:
- `429 Too Many Requests` (rate limit — always retry with `Retry-After` header respect)
- `500`, `502`, `503`, `504` (server errors — transient, retriable)
- `NetworkError` / connection reset (the fetch itself threw)
- `TimeoutError` (if the request never received a response — safe to retry)

Never retry:
- `400`, `401`, `403`, `404`, `409`, `422` — any 4xx except 429
- `408 Request Timeout` is ambiguous (server received the request) — do not retry by default

For `429`: parse the `Retry-After` response header (can be seconds or an HTTP date). Honor it. If absent, use exponential backoff with full jitter (not capped jitter — full jitter avoids thundering herd).

```typescript
// Full jitter: sleep = random(0, min(cap, base * 2^attempt))
const sleep = Math.random() * Math.min(cap, base * Math.pow(2, attempt));
```

**Warning signs:**
- Test logs showing multiple identical requests for a 400 response
- Rate limit errors escalating instead of recovering under load
- `Retry-After` header present in 429 responses but SDK ignoring it

**Phase:** Phase 1 (core HTTP client and retry logic).

### Pitfall 6: `@deepidv/core` Leaking Internal Types to Consumers

**What goes wrong:** `@deepidv/server` re-exports everything from `@deepidv/core` indiscriminately, including internal HTTP client types, configuration internals, and base classes not intended for public API. Consumers end up with `import { HttpClient } from '@deepidv/server'` working — and then depending on it.

**Why it happens:** Barrel `export * from '@deepidv/core'` in `packages/server/src/index.ts` — the lazy path.

**Consequences:** `@deepidv/core` internals become a de-facto public API. Refactoring core breaks external consumers. Types that should be internal appear in IDE autocomplete, polluting the developer experience.

**Prevention:**

`packages/server/src/index.ts` must enumerate exactly what is exported:

```typescript
// Explicit re-exports only — no wildcard from @deepidv/core
export { DeepIDV } from './deepidv';
export type { DeepIDVConfig } from './deepidv';
export type { SessionCreateParams, SessionCreateResult } from './types/sessions.types';
// ... one export per public surface
```

`@deepidv/core` should have `"private": true` or `"publishConfig": { "access": "restricted" }` in its `package.json` to signal it is not for direct install.

**Warning signs:**
- `import { HttpClient } from '@deepidv/server'` works in a consumer project
- VS Code autocomplete in a consumer shows internal class names
- `@deepidv/core` has a non-zero npm download count from external sources

**Phase:** Phase 1 (package structure). Enforce via ESLint `no-restricted-imports` if needed.

### Pitfall 7: pnpm Workspace — `workspace:*` Protocol Not Stripped at Publish

**What goes wrong:** pnpm uses `"@deepidv/core": "workspace:*"` in `@deepidv/server/package.json` to reference the local workspace package. If `@deepidv/server` is published without pnpm replacing that with the actual version number, consumers install a package that declares a dependency that doesn't exist on npm (`workspace:*` is not a valid npm version specifier).

**Why it happens:** Publishing with `npm publish` directly instead of through pnpm or a changesets pipeline. The `workspace:*` protocol is a pnpm-only concept.

**Consequences:** `npm install @deepidv/server` fails for consumers with `No matching version found for @deepidv/core@workspace:*`. Or, if `@deepidv/core` is not published at all (intentionally private), the dependency declaration in the published package is still wrong.

**Prevention:**

Always publish through `pnpm publish` (not raw `npm publish`) — pnpm automatically replaces `workspace:*` with the resolved version during `pnpm pack` and `pnpm publish`.

If `@deepidv/core` is internal-only and not published to npm, use `bundleDependencies` in `@deepidv/server/package.json` to bundle core's output into the server package, or use tsup's `noExternal` option to inline core code at build time:

```typescript
// packages/server/tsup.config.ts
export default defineConfig({
  noExternal: ['@deepidv/core'],  // Bundle core into server output
});
```

Use changesets for release management — it handles version coordination across workspace packages and generates proper changelogs.

**Warning signs:**
- `npm install @deepidv/server` in a clean project fails with version resolution errors
- Published `package.json` on npm contains `"workspace:*"` in dependencies
- CI publish step using `npm publish` directly instead of `pnpm publish`

**Phase:** Phase 1 (monorepo setup). Phase 5 (publishing pipeline) — must verify before first publish.

### Pitfall 8: Cloudflare Workers / Edge Runtime — No File System, No `Buffer` Global

**What goes wrong:** The SDK accepts file path strings as image input. Reading file paths requires `fs.readFile`, which does not exist in Cloudflare Workers or any edge runtime. Additionally, `Buffer` is a Node.js global that does not exist natively in Workers (Cloudflare Workers polyfill it partially, but it is not available in all edge runtimes).

**Why it happens:** It is natural to write `import { readFile } from 'fs/promises'` to handle path inputs. It works locally. It fails silently at deploy time to edge.

**Consequences:** Worker throws `ReferenceError: fs is not defined` or `ReferenceError: Buffer is not defined` at runtime. Since Workers don't have a build-time check for this, the error only appears after deploy. Edge consumers cannot use file path inputs at all.

**Prevention:**

1. File path support must be behind a runtime capability check — dynamically import `fs` only when it's available. Do this with a try/catch dynamic import, not a static import.

```typescript
async function filePathToBuffer(path: string): Promise<Uint8Array> {
  try {
    const { readFile } = await import('fs/promises');
    const buf = await readFile(path);
    return new Uint8Array(buf);
  } catch {
    throw new Error(
      'File path inputs are not supported in edge runtimes. ' +
      'Pass a Uint8Array or ReadableStream instead.'
    );
  }
}
```

2. Document this limitation clearly at the `image` parameter level in JSDoc.
3. Never use `Buffer` in `@deepidv/core`. Use `Uint8Array` throughout. `Uint8Array` is a web standard available everywhere. `Buffer` extends `Uint8Array` so Node.js `Buffer` instances pass `instanceof Uint8Array` checks — no conversion needed.

**Warning signs:**
- `ReferenceError: fs is not defined` in Cloudflare Workers logs
- `Buffer is not defined` errors in Deno or edge runtimes
- Tests pass in Node but fail in a Worker simulation

**Phase:** Phase 1 (core architecture). File path handling in Phase 2 — must not use static fs imports.

### Pitfall 9: `fetch` Body and Timeout — AbortController Cleanup on Success

**What goes wrong:** `AbortController`-based timeouts abort the request after N milliseconds. If the abort signal fires after the response headers arrive but before the body is fully consumed, the body stream is cancelled. The fetch resolves but `response.json()` (or `response.text()`) throws `AbortError`. Additionally: if the fetch succeeds normally, the timeout `setTimeout` handle is never cleared, keeping the Node.js event loop alive unnecessarily.

**Why it happens:** The standard pattern `setTimeout(() => controller.abort(), timeout)` is set-and-forget. Cleanup is an afterthought.

**Consequences:** In short-lived scripts (CLI tools, test suites), the process hangs for the full timeout duration after all requests complete. In long-running servers, accumulated uncleaned timers are a memory leak. On slow S3 uploads, a response might partially arrive then be aborted mid-body.

**Prevention:**

```typescript
async function fetchWithTimeout(url: string, init: RequestInit, timeoutMs: number): Promise<Response> {
  const controller = new AbortController();
  const timeoutId = setTimeout(() => controller.abort(), timeoutMs);
  try {
    const response = await fetch(url, { ...init, signal: controller.signal });
    return response;
  } finally {
    clearTimeout(timeoutId);  // Always clear — success AND failure
  }
}
```

Consume the response body outside the timeout window for S3 PUTs — S3 PUT responses are tiny (empty on success, small XML on error), so body consumption time is negligible and should be counted inside the timeout.

**Warning signs:**
- `jest --forceExit` required in test suite (event loop hanging)
- `AbortError` on response body consumption, not on the fetch itself
- Node.js process doesn't exit after SDK calls complete

**Phase:** Phase 1 (HTTP client), Phase 2 (S3 upload specifically — large body PUT to S3 must account for upload time in timeout calculation, not just connection time).

### Pitfall 10: Tree-Shaking Fails — Side Effects in Module Barrel

**What goes wrong:** `packages/server/src/index.ts` imports all modules at the top level, including ones with side effects (class property initializers, event emitter registration, module-level constants). Bundlers mark the entire package as having side effects and cannot tree-shake unused modules.

**Why it happens:** `export * from './modules/sessions'` in a barrel file is a common pattern. If any re-exported module has a side effect (even something as innocuous as `const DEFAULT_TIMEOUT = 30_000` at module scope is fine — but `EventEmitter` registration or `process.env` reads are not), the whole barrel gets marked unsafe to shake.

**Consequences:** Consumers who only use `client.face.compare()` pull in all session, document, and identity module code. Bundle size grows unnecessarily.

**Prevention:**

Add `"sideEffects": false` to each package's `package.json` to signal to bundlers that all files are side-effect free. If any file genuinely has side effects, list them explicitly: `"sideEffects": ["./src/polyfills.js"]`.

Audit each module: no top-level code that registers listeners, mutates globals, or reads environment variables. All initialization should be lazy (inside constructors or functions).

**Warning signs:**
- Webpack/esbuild bundle analysis shows session module included when only face module is used
- `"sideEffects"` key absent from `package.json`
- Module-level `process.env.NODE_ENV` reads in any file

**Phase:** Phase 1 (package.json setup), verified at Phase 5 (bundle analysis).

### Pitfall 11: Parallel Presigned Uploads — Error Propagation with `Promise.all`

**What goes wrong:** `face.compare()` uploads two images in parallel with `Promise.all([upload(source), upload(target)])`. If one upload fails, `Promise.all` rejects immediately (fail-fast) with the first rejection, but the second upload request may already be in-flight to S3. The presigned URL for the successful upload is now "burned" (used or expired) — and the SDK has no way to signal to the caller which image failed.

**Why it happens:** `Promise.all` is the natural tool for parallel uploads but its rejection semantics are not the same as sequential try/catch. The in-flight upload is not cancelled.

**Consequences:** Partial S3 state (one file uploaded, one not). The caller gets an error but cannot know which image caused the failure. Retrying the full `face.compare()` call works, but wastes one S3 upload.

**Prevention:**

1. Use `Promise.allSettled` to get both results, then inspect:

```typescript
const [sourceResult, targetResult] = await Promise.allSettled([
  uploadFile(source, sourcePresignedUrl),
  uploadFile(target, targetPresignedUrl),
]);
if (sourceResult.status === 'rejected') {
  throw new UploadError('source image upload failed', { cause: sourceResult.reason });
}
if (targetResult.status === 'rejected') {
  throw new UploadError('target image upload failed', { cause: targetResult.reason });
}
```

2. Attach `AbortSignal` to both uploads — if one fails, abort the other to avoid orphaned uploads.
3. Include which input (`source` or `target`) failed in the error message so developers know which image to check.

**Warning signs:**
- Unhandled promise rejection warnings in Node.js for S3 uploads
- `face.compare()` error messages that don't identify which image failed
- S3 storage growing with orphaned single-file uploads from failed pairs

**Phase:** Phase 3 (face.compare implementation).

### Pitfall 12: Zod Schema / TypeScript Type Drift

**What goes wrong:** The Zod schema for an input type is defined separately from (or before) the TypeScript type. As the API evolves, one is updated and the other is not. A field is required in the Zod schema but optional in the TypeScript type (or vice versa), creating a situation where TypeScript says a call is valid but Zod throws a runtime validation error, or vice versa.

**Why it happens:** Types and schemas are maintained in separate files and updated at different times. No enforcement that they stay in sync.

**Consequences:** Runtime `ZodError` on valid-looking code, or Zod passes invalid input that reaches the API and causes a confusing server-side error.

**Prevention:**

Always derive TypeScript types from Zod schemas — never maintain them independently:

```typescript
// Define schema first
const DocumentScanParamsSchema = z.object({
  image: z.union([z.instanceof(Buffer), z.instanceof(Uint8Array), z.string()]),
  documentType: z.enum(['passport', 'drivers_license', 'national_id', 'auto']).optional(),
});

// Derive type from schema — single source of truth
type DocumentScanParams = z.infer<typeof DocumentScanParamsSchema>;
```

Place the schema and its inferred type in the same file. Never write `interface DocumentScanParams { ... }` separately.

**Warning signs:**
- TypeScript shows no error but runtime throws `ZodError`
- A field appears required in one place and optional in another
- `type` and `schema` defined in different files that diverged

**Phase:** Phase 1 (establish the pattern in core), carried through all modules.

### Pitfall 13: Large File Upload Timeout — Separate Presign and Upload Timeouts

**What goes wrong:** A single global `timeout` setting is applied to all requests. The presign request to the deepidv API is fast (< 500ms). The S3 PUT of a large ID document image (could be 5-10MB for a high-res scan) takes much longer. If both use the same 30-second global timeout, a developer who sets `timeout: 5000` for API responsiveness breaks all image uploads on slow connections. A developer who sets a long timeout for uploads gets very slow error feedback on API failures.

**Why it happens:** Simple SDKs use one timeout. The presigned upload pattern has two very different network operations.

**Consequences:** Upload timeouts on real user data under normal conditions. Or unacceptably long hang times when the API is down.

**Prevention:**

Expose separate timeout settings:

```typescript
interface DeepIDVConfig {
  apiKey: string;
  timeout?: number;          // API request timeout (default: 30_000ms)
  uploadTimeout?: number;    // S3 upload timeout (default: 120_000ms)
}
```

Apply `timeout` to all deepidv API requests (presign, processing endpoints, session CRUD). Apply `uploadTimeout` to the S3 PUT specifically. Document both clearly.

For S3 upload specifically: the timeout should start from first byte sent, not connection establishment, to handle slow upload speeds on poor connections.

**Warning signs:**
- Users on mobile networks report intermittent upload failures
- 5MB image upload timing out with default settings
- Error message says "TimeoutError" with no indication it was the S3 upload step

**Phase:** Phase 2 (upload handler design).

### Pitfall 14: Error Wrapping Loses Original Stack Traces

**What goes wrong:** Custom error classes wrap raw `fetch` errors or Zod errors but discard the original `cause` and stack trace. When a developer reports a bug, the error they receive (`NetworkError: request failed`) has no inner cause pointing to the actual low-level error (`TypeError: Failed to fetch`), making debugging nearly impossible.

**Why it happens:** `throw new NetworkError(message)` is simpler than `throw new NetworkError(message, { cause: originalError })`.

**Consequences:** Support conversations start with "what's the actual error?" Every error that reaches a developer is opaque.

**Prevention:**

All custom errors use the `cause` option (Node 16.9+, all modern runtimes support it):

```typescript
class NetworkError extends DeepIDVError {
  constructor(message: string, options?: { cause?: unknown }) {
    super(message, options);
    this.name = 'NetworkError';
  }
}

// Usage:
try {
  await fetch(url);
} catch (err) {
  throw new NetworkError('Failed to reach deepidv API', { cause: err });
}
```

Zod parse errors should be wrapped similarly:
```typescript
const result = schema.safeParse(input);
if (!result.success) {
  throw new ValidationError('Invalid input', {
    cause: result.error,
    issues: result.error.issues,
  });
}
```

**Warning signs:**
- `console.log(error.cause)` returns `undefined` on any SDK error
- Error messages contain no information about the underlying cause
- Stack traces in bug reports only show SDK internals, not the original system error

**Phase:** Phase 1 (error class design). Pattern must be established before any other module is built.

## Minor Pitfalls

### Pitfall 15: `moduleResolution: node` in tsconfig Prevents Correct Package Resolution

**What goes wrong:** The SDK's own `tsconfig.json` uses `"moduleResolution": "node"` (the old default). When TypeScript resolves `@deepidv/core` as a workspace dependency during SDK development, it ignores the `"exports"` field in core's `package.json` and uses only `"main"`. This means the SDK builds fine against `.cjs` output but the ESM entrypoint is never validated by TypeScript during development.

**Why it happens:** `node` has been the default for years. `node16` and `bundler` are newer and require explicit opt-in.

**Prevention:**

Use `"moduleResolution": "bundler"` in `tsconfig.base.json` for all packages. This matches how tsup (esbuild underneath) resolves modules and correctly respects `"exports"` maps. Use `"module": "ESNext"` with `"moduleResolution": "bundler"` (they go together).

**Phase:** Phase 1 (root tsconfig setup).

### Pitfall 16: Base64 Input — Ambiguity Between Data URL and Raw Base64

**What goes wrong:** The SDK accepts `string` as an image input, with semantics of "file path or base64." If a developer passes a data URL (`data:image/jpeg;base64,/9j/4AAQ...`), the SDK interprets it as a file path (since it's a string), tries to read a file at that path, fails, and throws a `FileNotFoundError` — or worse, silently sends the literal string as the S3 upload body.

**Why it happens:** Data URLs and base64 strings look identical from a type system perspective. The distinction is entirely in the string's prefix.

**Prevention:**

In the string input handler, check for three patterns:
1. Starts with `data:image/` → data URL → strip the prefix, decode base64
2. Starts with `/` or `./` or `C:\` or is an absolute path → file path
3. Matches base64 character set with no path separators → raw base64

Make this detection explicit and documented. Consider rejecting data URLs with a clear error rather than silently handling them — forces callers to normalize input before passing to the SDK.

**Phase:** Phase 2 (upload handler input normalization).

### Pitfall 17: `changesets` Version Mismatch — `@deepidv/core` and `@deepidv/server` Out of Sync

**What goes wrong:** `@deepidv/server` declares `"@deepidv/core": "^1.0.0"` as a dependency. After internal refactoring, `@deepidv/core` is published at `1.1.0` with a breaking internal change, but `@deepidv/server` is not republished. Consumers on the old `@deepidv/server` now pull in the new `@deepidv/core` via semver range resolution and hit a runtime error.

**Why it happens:** Treating internal dependencies as safely semver-ranged when they are actually tightly coupled. Any change to core requires a coordinated server bump.

**Prevention:**

Use exact version pins (not ranges) between workspace packages in the published output: `"@deepidv/core": "1.0.0"`, not `"^1.0.0"`. Or, as noted in Pitfall 7, use `noExternal` in tsup to bundle core into server, eliminating the runtime dependency entirely. For a two-package monorepo, bundling is simpler than version coordination.

**Phase:** Phase 5 (publishing pipeline setup).
